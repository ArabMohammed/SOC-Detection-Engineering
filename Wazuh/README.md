# Rules and Decoders

## 1. Introduction

These two directories contains custom Wazuh decoders and rules developed to support log sources that are either not currently supported by the official Wazuh repository or whose existing decoders and rules are outdated and incompatible with newer product versions.

Currently supported log sources include:

- **Palo Alto Networks Firewalls v11.1**
  - User-ID
  - IP-Tag
  - SYSTEM
  - HIPMATCH
  - THREAT
  - TRAFFIC

- **F5 BIG-IP**
  - ASM (Application Security Manager) module

- **FortiAnalyzer v7.6.x**

These rules and decoders have been tested in production environments and are intended to provide improved log parsing and detection coverage compared to the default Wazuh content.

---

## 2. Installation and Usage

Before deploying the provided rules and decoders, ensure that none of your existing custom rules use the same rule IDs.

> **Important:** Some of the provided rules replace or extend the default Wazuh content. In our environment, we reused the same rule ID ranges as the official Wazuh rules because the original rules were replaced by our custom implementations.

### Install Decoders

Copy the decoder XML files to:

```bash
/var/ossec/etc/decoders/
```

### Install Rules

Copy the rule XML files to:

```bash
/var/ossec/etc/rules/
```

### Restart Wazuh Manager

```bash
systemctl restart wazuh-manager
```
---

## 3. Testing

Sample logs are provided in the `tests/` directory to help validate decoder and rule functionality.

You can use the Wazuh Logtest utility to verify that logs are correctly decoded and matched against the expected rules.

### Example

```bash
cat tests/log_example.log | /var/ossec/bin/wazuh-logtest
```

Verify that:

- The correct decoder is applied.
- All expected fields are extracted.
- The appropriate rule is triggered.

# Ingest Pipelines

This section contains custom Filebeat ingest pipeline modifications developed to enrich Wazuh alerts before they are indexed in Wazuh Indexer/OpenSearch.

---

# A. Calculate the Total CRScore per User

## Overview

The **Threat Weight** feature [https://docs.fortinet.com/document/fortigate/7.0.0/administration-guide/903511/threat-weight], defines multiple fields, including the `crscore` field, which can be used to aggregate and score threats based on user-defined severity levels.

This provides a practical way to calculate a cumulative threat score for each user based on their browsing activity and the websites they visit.

The challenge is that values extracted during the Wazuh decoding phase are stored as **strings**, which prevents them from being aggregated using functions such as `sum()` within OpenSearch Dashboards visualizations.

Two approaches can be used to solve this problem:

---

## Option 1: Using Scripted Fields

### Create a Scripted Field

1. Navigate to:

   ```text
   Dashboard Management → Index Patterns → wazuh-alerts-*
   ```

2. Open the **Scripted Fields** tab.

3. Click **Add Scripted Field**.

4. Configure the field:

   | Setting | Value |
   |----------|----------|
   | Name | crscorenumeric |
   | Language | Painless |
   | Type | Number |

5. Paste the following script:

```painless
def src = params._source;

if (src == null) return null;

def data = src.containsKey('data') ? src.data : null;
if (data == null) return null;

if (!data.containsKey('crscore') || data.crscore == null)
    return null;

def v = data.crscore;

try {
    return Integer.parseInt(v.toString());
} catch (Exception e) {
    return null;
}
```

6. Save the field.

7. Refresh the index pattern by clicking the **Refresh Fields** icon.

This ensures that the newly created field is recognized by the Wazuh Dashboard.

### Limitations

> Scripted fields are calculated at query time whenever a visualization references them. In environments with large datasets, this can significantly impact dashboard performance and increase visualization rendering times.

---

## Option 2: Using a Wazuh Ingest Pipeline (Recommended)

The Filebeat ingest pipeline enriches alerts generated by Wazuh Manager before they are forwarded to Wazuh Indexer.

Unlike scripted fields, ingest pipeline processing occurs only once during indexing, making it significantly more efficient for large environments.

### Step 1: Modify the Ingest Pipeline

Edit the following file:

```bash
/usr/share/filebeat/module/wazuh/alerts/ingest/pipeline.json
```

Add the following processor to the `processors` section:

```json
{
  "script": {
    "ignore_failure": true,
    "lang": "painless",
    "source": "if (ctx.containsKey(\"data\") && ctx.data != null) { if (ctx.data.containsKey(\"crscore\") && ctx.data.crscore != null && ctx.data.crscore != \"\") { try { def v = ctx.data.crscore.toString().trim(); ctx.data.crscorenumeric = (int) Double.parseDouble(v); } catch (Exception e) { ctx.data.crscorenumeric = 0; } } }"
  }
}
```

### Step 2: Load the Updated Pipeline

Execute:

```bash
filebeat setup --pipelines --modules wazuh
```

Restart Wazuh Manager:

```bash
systemctl restart wazuh-manager
```

### Step 3: Refresh the Index Pattern

Navigate to:

```text
Dashboard Management → Index Patterns → wazuh-alerts-*
```

Refresh the field list to ensure the new `crscorenumeric` field is recognized.

# B. Refine F5 ASM Attack Types

## Overview

F5 BIG-IP ASM logs often contain multiple attack types within the same event.

Examples include:

- Buffer Overflow
- Command Execution
- HTTP Parser Attack
- Forceful Browsing
- Detection Evasion
- Information Leakage

Many of these attack types are generic and frequently appear in almost every attack event. This reduces their usefulness when performing threat hunting, reporting, or dashboarding.

To improve attack classification, the following ingest pipeline script removes common generic attack types and retains only the most relevant attack categories.

### Processor

Add the following processor to the ingest pipeline:

```json
{
  "script": {
    "ignore_failure": true,
    "lang": "painless",
    "source": "if (ctx.containsKey(\"data\") && ctx.data != null && ctx.data.containsKey(\"attack_type\") && ctx.data.attack_type != null && ctx.data.attack_type != \"\") { def noise = [\"Abuse of Functionality\", \"Buffer Overflow\", \"HTTP Parser Attack\", \"Forceful Browsing\", \"Detection Evasion\", \"Information Leakage\", \"Other Application Activity\", \"Predictable Resource Location\"]; def parts = ctx.data.attack_type.splitOnToken(\",\"); def kept = new java.util.ArrayList(); for (int i = 0; i < parts.length; i++) { def v = parts[i].trim(); if (!noise.contains(v)) { kept.add(v); } } if (kept.size() > 0) { ctx.data.attack_type = String.join(\",\", kept); } else { ctx.data.attack_type = parts[0]; } }"
  }
}
```

### Example

#### Original Attack Types

```text
Buffer Overflow, Command Execution, HTTP Parser Attack
```

#### Processed Attack Types

```text
Command Execution
```

This modification improves attack categorization and produces more meaningful dashboards, reports, and threat-hunting queries.
